# ACS 4390 — How This Seminar Works

This course is different from most programming classes. Instead of lectures and step‑by‑step tutorials, this course is a **seminar**. A seminar is a class where students actively participate by researching, explaining ideas, and discussing technology together.

Your role in this course is not just to listen. You will:

- research technologies
- explain ideas to the group
- ask questions
- demonstrate experiments
- analyze trade‑offs in real tools

The goal is to help you think like a **professional frontend engineer** who can evaluate technologies and make architectural decisions.

---

# Weekly Structure

This class meets **twice per week on Zoom (Monday and Wednesday)**.

Each week focuses on a **theme** (for example: build tools, state management, performance, CSS systems, etc.). Students explore that theme from different perspectives.

Example:

**Week Theme: State Management**

Students might explore:

- Redux
- Zustand
- React Query
- Signals
- Server vs Client state

Everyone contributes something different, but the entire class discusses the **same overall problem**.

Weekly topics are meant to explore the **breadth of the frontend ecosystem**. Your **term project**, described later in this document, focuses on investigating **one topic in depth**.

---

# Monday vs Wednesday Sessions

Because we meet twice per week, each meeting has a different purpose.

## Monday — Technology Exploration

Monday sessions focus on learning about technologies and architectural ideas.

Typical structure:

| Time | Activity |
|-----|-----|
| Start | Preparation time |
| First hour | Lightning talks |
| Middle | Group discussion |
| End | Technology analysis and wrap‑up |

These sessions introduce new ideas and technologies.

---

## Wednesday — Engineering Lab

Wednesday sessions focus on **experiments, prototypes, and project progress**.

Typical structure:

| Time | Activity |
|-----|-----|
| Start | Project log check‑in |
| First hour | Prototype or demo presentations |
| Middle | Technical discussion |
| End | Project troubleshooting and next steps |

These sessions connect ideas from Monday discussions to real engineering work.

---

# Lightning Talks

Each week students will give short **lightning talks** related to the weekly theme.

Lightning talks are short explanations that introduce technologies or ideas to the class.

Typical structure:

1. **What problem does this tool solve?**
2. **How does it work?**
3. **Advantages**
4. **Trade‑offs or weaknesses**

Lightning talks are usually **5–8 minutes long** followed by a few minutes of questions.

These talks help the class explore the broader ecosystem of tools and technologies.

Important:

Lightning talk topics should **not be the same as your term project topic**. Weekly talks explore ideas broadly, while the term project investigates one topic deeply.

---

# Weekly Roles

Because this class has a small group, each student will take on a **role** during discussions. Roles rotate each week.

| Role | Responsibility |
|-----|-----|
| **Explainer** | Introduces the technology and explains how it works |
| **Critic** | Identifies weaknesses or trade‑offs |
| **Builder** | Demonstrates a prototype or experiment |
| **Historian** | Explains why the technology emerged and what came before it |
| **Connector** | Relates the technology to other tools or frameworks |

Roles help ensure that discussions are balanced and thoughtful.

---

# The Term Project

During the course you will complete a **term‑long research and prototype project**.

Your project should investigate a specific question related to frontend technology.

Examples:

- Does Vite significantly improve development startup time compared to Webpack?
- What trade‑offs do React Server Components introduce?
- How do different state management libraries affect application complexity?

The project should include:

1. Research
2. A small prototype or experiment
3. Analysis of results

Projects are not large applications. Instead they should be **small experiments that help you understand a technology deeply**.

---

# Midterm Progress Update

Around the middle of the course you will give a **midterm progress presentation**.

This short presentation will show:

- your research so far
- the prototype or experiment you are building
- early observations or challenges

The purpose of this presentation is to:

- receive feedback
- identify problems early
- refine your research direction

Midterm updates are typically **10 minutes long**.

---

# Final Project Presentation

At the end of the course each student will present their completed project.

Final presentations are longer than lightning talks and usually last **15–20 minutes**.

Typical structure:

1. The problem or question being investigated
2. Research and background
3. Prototype or experiment
4. Findings
5. Trade‑offs and conclusions

Final presentations will be scheduled **across several class sessions** so we have time for meaningful discussion.

---

# Project Logs

To help track progress, students will keep a **weekly project log**.

This log should briefly describe:

- What you learned
- What you built or tested
- What surprised you
- What you plan to try next

Example:

```
Week 3 Project Log

Topic
React Server Components

What I learned
RSC moves data fetching to the server and can reduce bundle size.

What I built or tested
Created a small Next.js demo using RSC.

What surprised me
Server round‑trips increased latency in some cases.

Next step
Measure bundle size differences.
```

These logs are short and usually take **5–10 minutes** to write.

---

# Weekly Preparation

Before class each student should prepare:

- **1 source** (article, documentation, talk, or tutorial)
- **1 insight** (something interesting you learned)
- **1 question** (something you want to understand better)

Example:

Source

> Blog post explaining React Server Components.

Insight

> React Server Components reduce bundle size by moving logic to the server.

Question

> How does this affect latency or server load?

This preparation usually takes **15–30 minutes**.

# Expectations for Participation

Because this is a seminar, participation is essential.

You are expected to:

- contribute to discussion
- present short explanations
- share experiments or demos
- ask thoughtful questions

In a small seminar every student should speak during each class session.

- share your screen

---

# The Goal of This Course

By the end of this seminar you should be able to answer questions like:

- Why does a technology exist?
- What problem does it solve?
- What are its trade‑offs?
- When should engineers use it?

These are the kinds of questions **experienced engineers ask when choosing tools**.

This course is designed to help you develop that level of thinking.
